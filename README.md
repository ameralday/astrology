# Vedic Astrology App — Prototype

A rough end-to-end prototype: enter birth data once, get an accurate Vedic (sidereal) natal
chart, and see a daily panchang-based overview on return visits.

- `/server` — stateless Node.js/TypeScript/Express backend. Computes charts with Swiss
  Ephemeris (Lahiri ayanamsa, whole-sign houses). No database — nothing is persisted.
- `/app` — Expo (React Native + TypeScript) app. Caches the user's birth profile locally
  on-device (AsyncStorage) so they don't re-enter it every visit; nothing leaves the device
  except the stateless calls to `/server`.

## Running it

### 1. Start the backend

```bash
cd server
npm install
npm run dev
```

This starts the API on `http://localhost:3000`. Quick sanity check:

```bash
curl http://localhost:3000/health
```

Run `npm test` in `/server` to run the nakshatra/rashi boundary unit tests.

### 2. Start the app

```bash
cd app
npm install
npx expo start
```

Scan the QR code with the **Expo Go** app on your phone (iOS or Android).

**Important:** your phone and computer must be on the same Wi-Fi network. The app
auto-detects your computer's LAN IP from the Expo dev server connection and points API
calls at `http://<that-ip>:3000`, so no manual configuration is needed — just make sure
the backend (step 1) is running before you open the app.

If you're using an iOS/Android **simulator/emulator** on the same machine instead of a
physical device, this should also work as-is; if it doesn't, fall back to
`npx expo start --tunnel`, which routes through a public tunnel instead of the LAN.

```
    ephemeris.ts           Swiss Ephemeris wrapper (julian day, planet positions,
                             sidereal ascendant, whole-sign houses, nakshatra/pada)

    constants/vedic.ts       27 nakshatras, 12 rashis, panchang name tables
    routes/chart.ts          POST /api/chart
    routes/panchang.ts       POST /api/panchang
    test/nakshatra.test.ts   boundary unit tests (pada/rashi edges)

/app
  App.tsx                    navigation stack + initial-route gate (has a saved profile?)
  src/screens/
    BirthDataScreen.tsx       name/date/time/place form, calls /api/chart, saves locally
    HomeScreen.tsx             greeting + today's panchang overview
    ChartScreen.tsx            full natal chart (ascendant, planets, houses)

    api.ts                     fetch wrappers for the backend

    geocode.ts                  offline place search + timezone lookup (bundled GeoNames data)
    config.ts                   auto-detects backend URL from the Expo dev server host
```

## API

`POST /api/chart` — body `{ date: "YYYY-MM-DD", time: "HH:mm", timezone: "Area/City", lat, lon }`
→ ascendant, whole-sign houses, and each planet's sign/nakshatra/pada/house/retrograde status.

`POST /api/panchang` — body `{ date, timezone, lat, lon }`
→ today's vara (weekday), tithi, nakshatra, yoga, and karana, computed for local noon on that date.

Both are pure functions of their input — nothing is written to disk or a database.

## Known simplifications (fine for a prototype, worth revisiting before a real launch)

- **Ephemeris precision**: uses Swiss Ephemeris's Moshier analytical model (no external data
  files needed) — accurate to about 1 arcsecond, well within what's needed for nakshatra/pada
  boundaries. Full JPL-derived `SwissEphemeris` data files can be swapped in later for even
  higher precision.
- **Panchang timing**: computed for local noon, not sunrise (the traditional reference point).
  Good enough for a "today's overview" card; a sunrise-based version would need a
  rise/transit calculation.
- **Geocoding**: place search runs entirely offline against a bundled GeoNames dataset
  (`app/src/data/cities.json`, generated by `app/scripts/build-cities.js`) — no live API call,
  no rate limit. Only covers cities with population > 15,000; see that script's comments for
  how to regenerate with a larger threshold if needed.
- **Licensing**: Swiss Ephemeris (and the `@swisseph/node` binding) is AGPL-3.0-licensed
  (see `server/LICENSE`) — since the backend is a network service, this requires its source to
  be published, which is why it's in this repo. Astrodienst also sells a commercial license as
  an alternative if the backend ever needs to be closed-source. Not a blocker for ad-supported
  use itself, just something to stay compliant with.
- **Auspicious-timing lookahead (muhurta)** is not built yet — deferred to a future phase.

## Deploying the backend (Google Cloud Run)

The backend is a stateless container - no database, no persistent volume needed. `server/`
has a multi-stage `Dockerfile` (compiles TypeScript in a build stage, ships only production
dependencies + compiled `dist/` in the runtime image) and already reads `process.env.PORT` and
binds `0.0.0.0`, which is what Cloud Run expects - no code changes needed to deploy it.

From the `server/` directory, with the [gcloud CLI](https://cloud.google.com/sdk/docs/install)
authenticated to your project:

```bash
gcloud run deploy astrology-backend --source . --region <your-region> --allow-unauthenticated
```

`--source .` uploads the directory and builds the container remotely via Cloud Build, so a
local Docker install isn't required. `--allow-unauthenticated` is needed since the app calls
this API directly from client devices (no server-to-server auth in front of it).

Once deployed, update `app/src/lib/config.ts` (or wherever `API_BASE_URL` is set for
production builds) to point at the Cloud Run URL instead of the LAN-IP auto-detection used
for local development.

If Cloud Run's cold starts (a few seconds on the first request after a scale-to-zero idle
period) become noticeable once real users are on it, `gcloud run services update
astrology-backend --min-instances=1` keeps one instance warm at all times, at a small ongoing
cost.
